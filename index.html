1. Sniffing at RSA

We will describe the basic mechanics of the RSA asymmetric encryption algorithm, developed by Adleman, Rivest, and Shamir in 1978. RSA is very widely used today for secure Internet communication, most notably SSL and HTTPS. The RSA cryptosystem involves modular arithmetic.

Key generation: To participate in the RSA cryptosystem, We must first generate a public and private key. We only need to do this once.

  • We Select two large prime numbers at random
    • Example: p = 11 (prime number), q = 29 (prime number)
  • We then compute them n = p × q
    • Example: n = 319 (11 * 29)
  • We then select two integers e and d
    • Example: e = 3 (random number), d = 187 (random number)
  • And as last we do a final check for all integers
    • Example: (3*187 mod ((11-1)*(29-1)) = 1)

Let's do the encryption and decryption ourself with message m = 289

Now we do it again but then with message m = 5555. Let's see what happens?

The reason why we get a different number (132) than 5555 is because we are using modular arithmetic. That means our value m (5555) cannot be greater then the remainder of m when divided by n (319).

Based on the outcome, you might conclude that RSA is useless for practical use. How we do we apply RSA in practice in order to circumvent its limitation?

We can use the symmetric cryptosystem or any other cryptosystem that handles the length of the message you want to encode. In practice we call this a hybrid cryptosystem one which combines the convenience of a public key cryptosystem (RSA) with the efficiency of a symmetric key cryptosystem. We can use RSA algorithm to encrypt a symmetric key and then the rest of the ciphertext is your encrypted message.

Now let's do it again but then with message m = 143 and let's encrypt it with the private key and decrypt it with the public key. So now we swap the roles of public and private keys.

Wow, it still works!

We swapped the roles of public and private keys. What is the use of this way of working?

By encrypting it with the private key, everyone with the public key will be able to decrypt it, which is not great for confidentiality but good for authenticity and integrity. Everyone will know where the encrypted message came from.

3. Certificates

Let's start by asking what exactly is a certificate?

Certificates are digital credentials that are used to verify the identity of individuals, computers, and other entities on a network. A Certificate binds a cryptographic key (public and private key) to a domain name, organisation identity, and location.

Now we know what a certificate is what is a self-signed certificate then?

A self signed certificate is a certificate that is not distributed by a certificate authority, but signed by one’s own private key.

4. Cryptographic hash functions

A cryptographic hash function must have 3 properties to define its security level. Once one collision has been found, the hash function is considered broken

A hash function must be able to withstand the following properties

  • 1. Pre-Image resistance: The hash value of a message must be different from that of another hash value.
  • 2. Second pre-image resistance: Same as pre-image attack but the small differences between the messages must always be different from each other in the hash.
  • 3. Collision resistance: Two different hash messages such as hash h1 and hash h2 should never be the same hash value. This is sometimes considered to as strong collision resistance.

6. TLS/SSL

TLS adds confidentiality and integrity to socket communication. It is the basis for HTTPS. In the network protocol stack, TLS is part of the transport layer and uses TCP or UDP. HTTP is in the application layer. SSL is the predecessor of TLS and is considered insecure.

Capturing packets in an SSL session

Server Hello Record
Now we gonna locate the Server Hello SSL record. Wich cipher suite Does this record specify? What are the algorithms in the chosen cipher suite?

  • Public-key (asymmetric) algorithm: RSA
  • Symmetric-key algorithm: AES
  • Hash algorithm: SHA256

Does this record include a nonce? If so, how long is it? What is the purpose of the client and server nonces in SSL?

The purpose of a nonce is to prevent a replay attack on the server, thanks to the timestamp in the nonce the server will know wich was the last packet the client sent to the server.

Does this record include a session ID? What is the purpose of the session ID?

The Server Hello Record does contain a session ID, the purpose of it is to be a identifier for a resuming client session. The client can use this ID to continue in his previous session.

Does this record contain a certificate, or is the certificate included in a separate record. Does the certificate fit into a single Ethernet frame?

Client Key Exchange Record
Let's Locate the client key exchange record. Does this record contain a pre-master secret? What is this secret used for? Is the secret encrypted? If so, how? How long is the encrypted secret?

The client key exchange record does not contain a pre-master secret. However, since the selected server key exchange was RSA, a pre-master secret was probably generated from a diffie-hellman exchange protocol. The pre-master secret is the value that you receive from a key exchange, the client generates a pre-master secret and some random bytes (PRNG). The client then encrypts the pre-master secret with the server’s public key (received during the certificate message). The encrypted key is then send to server as a client key exchange message. The server then decrypts the encrypted key using its own private key, at this point both parties know the pre-master secret. both the client and the server then compute the master secret by calculating the client and the server nonces (random) with the pre-master secret. With this both sides now know the master secret. Since Wireshark does not show the pre-master secret in the field description, it must be within the client key exchange record. The overall length of the record is 66 bytes.

Change Cipher Spec Record (sent by client) and Encrypted Handshake Record
Now let's look in the Change Cipher Spec Record. What is the purpose of the Change Cipher Spec record? How many bytes is the record in your trace?

The change cipher spec record is sent by both the client and the server to notify the receiving end to encrypt the next incoming SSL records of the contents with the current or new cipher suite.

Wat is being encrypted? In the encrypted handshake record and how?

The encrypted handshake record consist of all the handshake messages from both client and the server. The encrypted handshake record is always sent after the change cipher spec record from both sides.

Does the server also send a change cipher record and an encrypted handshake record to the client? How are those records different from those sent by the client?

When the client sent its last encrypted handshake message to the server the message is then turned into a "finished message". This means that no one tampered with the encrypted handshake and it proves that we know the master secret. The server then starts decrypting the encrypted handshake message and it sends out a change cipher spec record. After the change cipher spec have been send, the server then sends a finished message that contains the decrypted message from the client. This proves to the client that the server was able to successfully decrypt the encrypted handshake message.

Application Data
We are finally reaching the end, how is the application data being encrypted? Do the records containing application data include a MAC? Does Wireshark distinguish between the encrypted application data and the MAC?

The application data contains the actual data of the messages. The data is encrypted with an symmetric encryption algorithm. The records do contain the Message Authentication Code (MAC) but Wireshark does not distinguish between the encrypted application data.

For assignment 4 we were given the task to finish implementing the Scrooge Coin, more specifically 2 functions in the transaction handle class. The full code can be found here

isValidTx()

The first method is the isValidTx(); this methode validate whether or not the given transaction is valid. There are 5 requirments for it to pass.

This method starts by creating a ArrayList variable to hold all the known Unspend Transactions (UTXO). It also creates 2 variables to hold the inputs and outputs.

It then starts by looping through the inputs to validate them. A variable is then made to hold the pool to check and then validate them to see if it has already been spend, if it is not already spent it wil be added to the spend pool and the inputs sum. Lastly it wil validate whether or not the signature is valid.

The last step is to validate the outputs. For the outputs to be valid, it must not be negative. If everything checks out it will validate if the input is greater or equal to the output and then return true.

handleTxs()

The second method we need to implement for this assignment is the handleTxs(); This methode is responsible for handling the transaction. This method start with looping through the given transaction. It uses the isValidTx(); to validate each transaction, and remove all inputs from the UTXO pool and add the outputs in the pool. After that it will return all the transaction that are valid back in an array.
